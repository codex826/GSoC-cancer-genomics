# Similarity Maps for Cancer Genomics in cBioPortal

## About This Project

Cancer research today produces massive genomic datasets — thousands of samples across dozens of studies, each with mutation profiles, copy number alterations, clinical annotations, and more. While cBioPortal excels at querying and filtering this data, there has been no way to **visually explore the inherent structure** of these datasets: Which samples look genomically similar? Do samples from different cancer types cluster together? How does a specific gene mutation relate to the broader population of samples?

**Similarity Maps** solves this by bringing UMAP (Uniform Manifold Approximation and Projection) embeddings directly into cBioPortal's Study View. A UMAP embedding compresses high-dimensional genomic data into a 2D scatter plot where **proximity = similarity**. Samples that are genomically alike appear close together; outliers and subtypes separate naturally. Researchers can then color these points by cancer type, gene mutation status, copy number alterations, or any clinical attribute — making it immediately clear how molecular features relate to population structure.

This implementation provides the **complete full-stack infrastructure** needed to store, serve, and visualize embedding data within cBioPortal — from the database all the way to an interactive, high-performance WebGL visualization.

## What Was Built

### 1. Database Layer

A new `sample_embedding` table was added to cBioPortal's MySQL schema (in `cgds.sql` and a migration script). Each row stores a single sample's coordinates in a named embedding space:

```
sample_embedding
├── cancer_study_id   → links to the study
├── sample_stable_id  → the sample identifier
├── embedding_name    → e.g., "umap", "pca"
├── x                 → horizontal coordinate
├── y                 → vertical coordinate
└── z                 → optional third dimension
```

This design allows multiple independent embedding spaces per study (e.g., RNA-seq UMAP, mutation signature UMAP) to coexist in the same database.

---

### 2. Java Backend – Domain, Persistence & API

A complete Java persistence stack was built following cBioPortal's existing MyBatis patterns:

- **`SampleEmbedding.java`** — A lightweight Java record (domain model) representing one coordinate point.
- **`SampleEmbeddingRepository.java`** + **`SampleEmbeddingMyBatisRepository.java`** — Repository interface and its MyBatis implementation, with SQL queries in `SampleEmbeddingMapper.xml`.
- **`SampleEmbeddingService.java`** — Service layer that retrieves embeddings by study ID and embedding name.
- **`SampleEmbeddingController.java`** — A Spring Boot REST controller exposing:

  ```
  GET /api/studies/{studyId}/embeddings/{embeddingName}
  ```

  This returns a JSON object mapping each sample ID to its `{x, y}` coordinates, making it consumable directly by the frontend.

---

### 3. Data Ingestion Tool

**`SampleEmbeddingLoader.java`** is a standalone Java utility that reads a standard TSV file (columns: `SAMPLE_ID`, `UMAP_1`, `UMAP_2`, optionally `UMAP_3`) and bulk-inserts the embeddings into the database. This enables teams to load precomputed embeddings (generated by Python tools like UMAP-learn or Zarr-based pipelines) directly into a running cBioPortal instance.

---

### 4. Frontend – Interactive Visualization

The frontend integrates the complete Deck.GL-based visualization suite (from the `add-umap` PR branch):

**`EmbeddingDeckGLVisualization.tsx`** — The core component. It uses WebGL via Deck.GL to render scatter plots with tens of thousands of points at 60fps. Features include:
- Smooth pan and zoom
- Lasso and rectangle selection (selected samples sync with other Study View filters)
- Pinnable tooltips showing sample ID, patient ID, and annotated values
- Dynamic coloring by cancer type, clinical attributes, or gene-level alterations

**`EmbeddingsTab.tsx`** (modified) — The Study View tab that orchestrates everything:
- Fetches embedding coordinates from the new `/api/studies/{studyId}/embeddings/umap` endpoint
- Transforms the response into the visualization's data format
- Wires MobX reactive state so coloring, selection, and URL parameters stay in sync
- Supports coloring by any clinical attribute or gene (with mutation, CNA, and structural variant annotation)

**Feature flag** (`featureFlags.ts`) — The `EMBEDDINGS` flag must be enabled to show the Similarity Maps tab, allowing controlled rollout.

**Dependencies** (`package.json`) — Added `@deck.gl/core`, `@deck.gl/layers`, `@deck.gl/react`, and `@deck.gl/widgets` for WebGL-accelerated rendering.

**TypeScript config** (`tsconfig.json`) — Fixed a path mapping bug where the wildcard `"*": ["src/*"]` was intercepting resolution of external packages like `react` and `mobx`, causing IDE-wide type errors.

## Why This Is Useful

| Use Case | How Similarity Maps Helps |
|---|---|
| **Subtype discovery** | Researchers can visually identify novel sample clusters not captured by existing classifications |
| **Outlier detection** | Samples far from their labeled cluster may indicate mislabelling or unusual biology |
| **Gene impact analysis** | Color the map by a gene — instantly see whether mutated samples cluster or scatter |
| **Cross-study comparison** | Embeddings spanning multiple studies expose shared biology across datasets |
| **Cohort selection** | Lasso-select a cluster → those samples become the active cohort for downstream analysis in other Study View tabs |

The feature is designed to scale: the Deck.GL renderer handles 100,000+ points without degradation, and the backend query is indexed on `(cancer_study_id, embedding_name)` for fast retrieval regardless of database size.
